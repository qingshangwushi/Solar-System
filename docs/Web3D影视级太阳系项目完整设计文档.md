# 1. 项目概述

## 1.1 项目定义

本项目建设一套运行于桌面浏览器中的、可完整离线部署的影视级太阳系真实模拟与交互展示系统。

系统以公开天文数据、星历数据、行星表面影像、高程数据和物理参数为基础，在浏览器中动态呈现太阳、八大行星、全部已命名天然卫星、矮行星、代表性小行星与彗星，以及小行星带、特洛伊小行星群、柯伊伯带、奥尔特云、太阳风和日球层等扩展结构。

系统需要同时满足四类真实性：

1. **天文真实性**：天体位置、速度、轨道、自转、轴向和事件关系可追溯到公开数据或明确的科学模型。
2. **视觉真实性**：S 级核心天体达到实时影视级近景表现，具备高质量材质、大气、云层、环系、太阳表面和空间环境效果。
3. **尺度真实性**：提供真实比例模式，并以明确标识的增强模式解决真实太阳系尺度下难以观察的问题。
4. **交互真实性**：用户能够在太阳系尺度、行星系统尺度、低轨尺度和部分地表尺度之间连续浏览，并控制模拟时间。

## 1.2 产品定位

本项目不是单一的三维网页，也不是天文科研软件，而是以下能力的统一体：

- 影视级太阳系视觉展示；
- 高精度天文时间与轨道演示；
- 面向普通用户的自由探索；
- 结构化科普数据查询；
- 预制主题巡航与天文事件导航；
- 可在无公网环境中长期运行的纯前端静态产品。

## 1.3 核心价值

- 将太阳系宏观结构、天体运动和近景地貌整合到同一个连续 Web3D 场景中；
- 在视觉增强与科学真实性之间建立可解释、可切换的边界；
- 不依赖云端服务即可运行，适用于个人电脑、内网和封闭环境；
- 通过统一的数据与资产管线持续扩充天体目录，而不把天体数量硬编码在程序中；
- 形成可复用的天文模拟、巨尺度坐标和行星渲染技术底座。

# 3. 建设目标与成功标准

## 3.1 建设目标

### G1：建立可信的太阳系模拟底座

在 1900—2100 年范围内，根据模拟时间计算核心天体的位置、速度、姿态、昼夜关系和相互遮挡关系。对于数据不足的天体，必须明确降低精度等级，不伪造高精度结果。

### G2：达到实时影视级核心天体效果

太阳、八大行星和月球必须具备独立渲染方案，不能使用同一套通用球体材质简单替换纹理。

### G3：实现跨数量级连续导航

用户应能够从太阳系全景连续进入行星系统、卫星系统、低轨和可支持的地表层级，过程中不出现不可接受的坐标抖动、裁剪错误或尺度突变。

### G4：完整离线运行

系统核心功能、数据、字体、模型、纹理、着色器、科普内容和巡航配置均不得依赖公网。

### G5：建立可持续扩展的数据与资产体系

天体目录、卫星数量、星历文件、科普内容和资产版本应通过静态数据清单驱动，不因新增天体修改核心引擎代码。

## 3.2 成功标准

项目被认为成功，至少需要满足：

- 用户可在真实时间或任意指定时间观察太阳系运行；
- 主要星体的位置和事件结果能够通过自动测试与外部基准对照；
- 真实比例与增强比例切换透明、可解释；
- S 级天体近景表现达到项目视觉验收基线；
- 地球、月球、火星可全球连续下降和低空浏览；
- 全部已命名天然卫星能够被搜索、定位并查看数据；
- Chrome/Edge 在正式硬件基线上连续稳定运行两小时以上；
- 断开公网后，所有验收功能仍可完整使用；
- 任意视觉增强内容均不会被错误标识为真实观测结果。

# 4. 项目范围与边界

## 4.1 纳入范围

### 自然天体

- 太阳；
- 水星、金星、地球、火星、木星、土星、天王星、海王星；
- 截至当前数据基线版本已确认并正式命名的全部天然卫星；
- 已公认矮行星；
- 具有代表性的候选矮行星；
- 具有典型科学和视觉价值的小行星；
- 具有典型科学和视觉价值的周期彗星和长周期彗星。

### 空间结构

- 主小行星带；
- 木星特洛伊小行星群及其他典型共轨群体；
- 柯伊伯带；
- 离散盘的概念性表现；
- 奥尔特云科学推断可视化；
- 黄道尘埃和黄道光；
- 太阳风；
- 日球层、日球层顶和电流片；
- 行星磁层和极光。

### 天文现象

- 昼夜变化；
- 日食和月食；
- 行星凌日；
- 天体掩星；
- 行星合、冲和最大距角；
- 近日点和远日点；
- 卫星食、凌越和投影；
- 行星环投影和遮挡。

## 4.2 明确不纳入范围

- 人造卫星、空间站、探测器、火箭和航天任务轨迹；
- 太阳系形成、太阳生命周期和数百万年以上长期演化；
- 航天导航级、科研观测级或任务设计级精度承诺；
- N 体实时长期积分作为默认运行方式；
- 用户账户、登录、云端同步、收藏和历史记录；
- 用户创作镜头、编辑巡航、导出项目文件或多人协作；
- 后端服务、数据库、内容管理系统和远程配置平台；
- 实时太阳活动、实时云图或实时外部科学数据流；
- 地球城市建筑、道路、植被和社会活动模拟；
- 星球表面开放世界、生存玩法或碰撞物理游戏；
- 移动端专项适配；
- VR、AR、WebXR；
- 音乐、旁白和操作音效；
- 浏览器本地偏好持久化；
- 基于用户地理位置的本地天象提醒；
- 对未知或无数据天体表面的“真实还原”宣称。

## 4.3 关键真实性边界

“严格符合现实”在本项目中定义为：

1. 有公开数据的部分，优先使用公开数据；
2. 有可靠数值模型的部分，按模型计算；
3. 数据缺失时，使用明确标识的统计模型或程序化示意；
4. 不将影视增强误写为观测事实；
5. 不对超出数据覆盖范围的结果给出虚假精度；
6. 不宣称获得 NASA、JPL、IAU、ESA、USGS 或天文学专家的官方认证。

# 5. 用户角色与核心场景

## 5.1 用户角色

### U1：普通探索用户

目标是直观地浏览太阳系、查找星体、观察运动和获取基础知识。

### U2：天文爱好者

目标是选择具体日期、查看轨道、观察事件、比较真实尺度并检查天体参数。

### U3：教学展示人员

目标是使用预制巡航和科普卡片完成稳定、可重复的演示。

### U4：视觉体验用户

目标是在纯净界面中体验影视级太阳、行星、大气、环系和空间环境。

### U5：项目运维人员

目标是安装离线包、验证资源完整性、更新版本、定位兼容性和性能问题。

## 5.2 核心使用场景

1. 打开系统后同步当前时间，查看真实太阳系相对位置；
2. 搜索“土星”，平滑飞行到土星并观察环系和主要卫星；
3. 切换真实比例，观察地球在太阳系尺度中的实际大小；
4. 选择 2024 年某次日食，自动跳转至事件极大时刻和推荐观察视角；
5. 从木星跳转至木卫一，再跳转至木星卫星系统全景；
6. 从地球高轨连续下降到地表低空；
7. 播放“太阳系尺度对比”预制巡航；
8. 在普通集成显卡设备上自动进入标准模式并保持可用；
9. 在无公网局域网中通过本地 HTTPS 服务运行影视级模式；
10. 通过诊断面板确认 WebGPU 状态、资源包版本和当前显存预算。

# 6. 产品模式设计

## 6.1 自由探索模式

- 默认模式；
- 支持鼠标、键盘和触控板；
- 支持目标锁定、自由飞行、轨道环绕和目标跟随；
- 支持时间控制和尺度切换；
- UI 信息量适中。

## 6.2 科学观察模式

- 默认使用真实比例或明确显示增强倍率；
- 显示参考系、时间尺度、距离、速度和精度等级；
- 显示轨道、轴向、母星关系和事件几何；
- 弱化或关闭 R4 影视增强；
- 禁止将程序化对象显示为真实地貌。

## 6.3 科普浏览模式

- 强化文字和数据卡片；
- 提供术语解释、结构化章节和数据对比；
- 简化专业参数；
- 保留时间和场景交互，但降低信息复杂度。

## 6.4 影视观赏模式

- 隐藏大部分 UI；
- 开启完整影视级渲染效果；
- 允许使用增强尺度；
- 支持预制巡航；
- 不改变底层天体状态，只改变视觉表达。

## 6.5 事件观察模式

- 由事件入口进入；
- 自动设置时间、目标、相机和必要图层；
- 显示事件阶段和几何关系；
- 支持退出后返回自由探索。

# 7. 完整功能需求

## 7.1 启动与能力检测

| 编号 | 需求 |
|---|---|
| FR-BOOT-001 | 系统启动时检测浏览器、操作系统、WebGPU、WebGL 2、纹理压缩格式、最大纹理尺寸和 GPU 限制。 |
| FR-BOOT-002 | 系统执行短时基准测试，根据 GPU 时间、CPU 时间和帧率给出推荐画质。 |
| FR-BOOT-003 | 若 WebGPU 不可用，自动进入 WebGL 2 标准模式。 |
| FR-BOOT-004 | 若核心资源缺失或校验失败，显示缺失资源包和文件路径。 |
| FR-BOOT-005 | 启动过程显示分阶段进度：核心程序、星历、基础天体、当前目标资产。 |
| FR-BOOT-006 | 不读取或写入浏览器持久化偏好。 |

## 7.2 时间模拟

| 编号 | 需求 |
|---|---|
| FR-TIME-001 | 支持系统当前 UTC 时间同步。 |
| FR-TIME-002 | 支持输入 1900—2100 年内任意日期和时间。 |
| FR-TIME-003 | 支持暂停、继续、正向、反向和单步推进。 |
| FR-TIME-004 | 支持秒、分钟、小时、日、月、年等多级时间倍率。 |
| FR-TIME-005 | 高倍率时按目标时刻直接计算状态，不使用帧累计近似。 |
| FR-TIME-006 | 时间变化后，所有天体位置、自转、阴影和事件状态同步更新。 |
| FR-TIME-007 | 超出范围时阻止继续并说明高精度范围边界。 |
| FR-TIME-008 | 对未来尚未确定的闰秒使用项目内的预测策略，并显示时间不确定性说明。 |

## 7.3 天体模拟

| 编号 | 需求 |
|---|---|
| FR-ASTRO-001 | 太阳、八大行星和月球使用高精度星历数据。 |
| FR-ASTRO-002 | 主要卫星优先使用可用的高精度卫星星历。 |
| FR-ASTRO-003 | 其他卫星按来源采用星历、数值拟合轨道或平均轨道根数。 |
| FR-ASTRO-004 | 平均轨道根数不得被标识为高精度星历。 |
| FR-ASTRO-005 | 每个天体状态包含位置、速度、姿态、角速度和精度等级。 |
| FR-ASTRO-006 | 轨道线通过指定时间区间采样状态生成。 |
| FR-ASTRO-007 | 所有天体使用唯一稳定标识，不依赖显示名称关联。 |
| FR-ASTRO-008 | 天体目录随离线数据包更新，不要求修改引擎代码。 |

## 7.4 尺度与显示

| 编号 | 需求 |
|---|---|
| FR-SCALE-001 | 提供真实比例模式。 |
| FR-SCALE-002 | 提供增强展示模式。 |
| FR-SCALE-003 | 分别显示距离倍率、星体半径倍率和卫星系统倍率。 |
| FR-SCALE-004 | 尺度切换必须平滑过渡，不允许突然跳变。 |
| FR-SCALE-005 | 在增强模式下，轨道和标签不得造成真实数据误读。 |
| FR-SCALE-006 | 真实比例模式下，标签可采用屏幕空间增强，但天体实体不得放大。 |

## 7.5 相机与导航

# 8. 非功能需求

## 8.1 性能

- 影视级模式以 RTX 4070/RX 7800 XT、32 GB 内存、1440p 为基线；
- 常规场景平均帧率目标 60 FPS；
- 高复杂度近景不应长期低于 45 FPS；
- 标准模式在 1080p 下不低于 30 FPS；
- UI 线程长任务应尽量小于 50 ms；
- 时间切换后核心天体状态更新应在 100 ms 内可见；
- 已安装资源包不设总容量上限，但单会话活跃 GPU 资源必须受预算控制；
- 连续运行两小时不出现持续内存增长。

## 8.2 稳定性

- GPU 设备丢失时尝试重建渲染器；
- 单个资源损坏不得导致整个应用白屏；
- Worker 崩溃后可重新初始化；
- 星历数据错误必须进入安全失败，而不是继续输出异常位置；
- 预制巡航中断后可返回自由探索。

## 8.3 可维护性

- 天文计算、渲染、UI 和资产管线独立；
- 数据格式带版本号；
- 所有天体资产通过 Manifest 声明；
- 渲染后端通过统一接口封装；
- 所有关键算法具备基准测试数据；
- 不在 React 状态树中保存逐帧天体位置。

## 8.4 可解释性

- 所有精度降级可被用户识别；
- 所有增强尺度可被用户识别；
- 所有程序化科学推演可被用户识别；
- 不以超出来源精度的小数位展示数据。

## 8.5 可部署性

- 单机、内网服务器和离线工作站均可部署；
- 正式包包含启动、停止、校验和诊断脚本；
- 不要求安装数据库或运行应用服务器；
- Windows 10/11 部署过程应可由非开发人员完成。

## 9.2 推荐技术栈

### 应用层

- React；
- TypeScript 严格模式；
- Vite；
- shadcn/ui；
- 轻量状态管理器，仅保存应用级低频状态；
- CSS Variables 统一界面主题；
- Web Worker 通信封装。

### 三维引擎层

- Three.js；
- WebGPURenderer 作为高质量路径；
- WebGLRenderer 作为兼容路径；
- TSL/WGSL 节点或独立 WGSL 模块；
- GLSL 兼容着色器；
- 自研渲染图、质量策略和资源管理器。

### 天文内核

- Rust；
- WebAssembly；
- Web Worker；
- 构建阶段使用 NAIF SPICE 工具处理星历；
- 运行时使用自定义紧凑二进制和切比雪夫插值；
- 事件算法使用数值求根和几何判定。

### 数据和资产工具链

- Python 用于数据 ETL、清洗、索引和报表；
- NAIF SPICE Toolkit 用于星历和姿态数据读取；
- GDAL 和行星数据处理工具用于投影和瓦片；
- Blender 用于模型处理和资产制作；
- KTX2/Basis Universal 用于纹理压缩；
- glTF/GLB 用于模型交换；
- Meshopt 用于几何压缩；
- SHA-256 用于完整性校验。

## 9.3 关键架构原则

1. React 不参与逐帧天体状态更新；
2. 天文内核不引用 Three.js 类型；
3. 渲染引擎不自行推算天体轨道；
4. 所有运行时数据通过版本化 Schema 交付；
5. WebGPU 和 WebGL 2 共享语义，不要求像素级完全一致；
6. 高精度数据优先，缺失时按规则降级；
7. 大文件必须按需加载，禁止启动时全量读取；
8. 建设阶段的联网数据加工与运行阶段的完全离线严格分离。

# 11. 天文时间系统

## 11.1 时间表示

应用层显示：

- 公历日期；
- UTC；
- 当前时间倍率；
- 当前时间精度说明。

内核使用：

- Julian Date；
- UTC；
- TAI；
- TT；
- TDB。

## 11.2 时间转换链

```text
用户输入 UTC
   ↓
闰秒表转换为 TAI
   ↓ + 32.184 s
TT
   ↓ 周期修正
TDB
   ↓
星历插值与事件计算
```

## 11.3 闰秒策略

闰秒由国际机构按实际地球自转情况发布，未来闰秒不能提前准确确定。因此：

- 离线包包含发布时最新闰秒表；
- 当前日期之前使用正式闰秒；
- 未来日期在未发布新闰秒时采用“最后已知表 + 预测 Delta T”策略；
- 对未来 UTC 事件时间显示“预测时间转换”说明；
- 内部轨道计算优先基于连续时间尺度，避免 UTC 跳秒导致轨道不连续；
- 数据包升级时可替换闰秒和 Delta T 数据，不修改内核。

## 11.4 时间倍率

建议档位：

- -1 年/秒；
- -30 日/秒；
- -1 日/秒；
- -1 小时/秒；
- -1 分钟/秒；
- 实时反向；
- 暂停；
- 实时；
- 1 分钟/秒；
- 1 小时/秒；
- 1 日/秒；
- 30 日/秒；
- 1 年/秒。

高倍率下只保证天体状态正确，不保证高速过程中的所有短时视觉现象都被逐帧完整采样。对于食等短时事件，应使用事件播放模式自动控制合适倍率。

# 12. 坐标、参考系与数值精度

## 12.1 科学计算坐标

内核统一采用双精度浮点。

核心参考系：

- 太阳系质心惯性系；
- 日心惯性系；
- 行星质心系；
- 天体固连系；
- 地表局部东-北-天坐标系；
- 观察者相对系。

每个状态必须携带参考系标识，不允许把不同参考系向量直接相加。

## 12.2 渲染坐标

GPU 不直接接收以千米计的太阳系绝对坐标。采用：

1. CPU 双精度科学坐标；
2. 当前焦点局部参考系；
3. 相机相对坐标；
4. 必要时使用高低位拆分；
5. GPU 使用 32 位浮点渲染。

## 12.3 浮动原点

触发条件：

- 相机距局部原点超过阈值；
- 切换目标；
- 从太阳系层进入行星层；
- 从行星层进入地表层。

重定位过程要求：

- 科学坐标不变；
- 仅重建渲染相对矩阵；
- 粒子和轨道缓存同步变换；
- UI 距离显示不受影响；
- 不产生可见跳动。

## 12.4 深度策略

按场景选择：

- 反向 Z；
- 浮点深度纹理；
- 对数深度；
- 多相机分层合成；
- 近景和远景独立渲染。

不建议始终依赖单一对数深度方案，因为透明体、后期处理和部分 GPU 路径可能产生兼容问题。

## 13.1 天体目录更新原则

“全部已命名天然卫星”使用数据快照实现，不在文档中固定数量。

- 星历覆盖范围；

1. NAIF Object ID；
2. JPL/Horizons ID；
3. MPC 编号；
4. 项目内部命名空间 ID。

显示名称不得作为主键。

## 13.3 核心天体数据结构

每个天体最少包含：

- `bodyId`；
- `type`；
- `parentBodyId`；
- 中文名；
- 英文正式名；
- 别名；
- 发现或命名信息；
- 平均半径或三轴尺寸；
- 质量或 GM；
- 平均密度；
- 反照率；
- 自转周期；
- 轨道周期；
- 极轴参数；
- 本初子午线模型；
- 星历提供器；
- 星历有效期；
- 精度等级；
- 资产等级；
- 内容引用；
- 来源列表。

## 13.4 数据精度等级

| 等级 | 定义 |
|---|---|
| P0 | 未知或仅有名称，不能计算可靠轨道 |
| P1 | 平均轨道根数或低精度拟合，仅适合科普显示 |
| P2 | 有限时间段数值星历或较完整摄动模型 |
| P3 | 高精度 SPK/等效星历，适合项目科学模式 |
| P4 | 核心天体高精度星历和姿态数据，经过自动基准对照 |

UI 不必显示 P0—P4 代码，但需要显示“高精度、计算模型、近似轨道、数据不足”等可理解文本。

# 14. 星历与轨道模拟引擎

## 14.1 总体策略

采用“高精度星历为主，分级模型为辅”的混合方案。

### 核心天体

太阳、八大行星和月球使用 JPL 行星与月球星历数据，经构建管线转换为浏览器友好的分段切比雪夫系数。

### 主要卫星

优先使用 JPL/NAIF 提供的卫星 SPK 文件。对于不同卫星系统使用独立 Kernel，并记录覆盖时间。

### 资料有限卫星

按优先级选择：

1. 高精度 SPK；
2. 数值积分拟合系数；
3. 带摄动项的轨道模型；
4. 平均轨道根数；
5. 只显示目录，不提供任意时间高精度位置。

### 小行星和彗星

- A 级代表对象使用独立星历；
- B 级对象使用轨道根数和历元传播；
- C 级群体使用统计轨道分布和实例化。

## 14.2 构建阶段星历处理

```text
公开 SPK/PCK/文本数据
      ↓
来源许可和版本登记
      ↓
SPICE 读取和采样验证
      ↓
按 1900—2100 时间范围裁剪
      ↓
生成分段切比雪夫系数或紧凑采样块
      ↓
误差分析
      ↓
生成浏览器二进制 + 索引 + 校验报告
```

## 14.3 运行时插值

运行时根据：

- bodyId；
- TDB 时间；
- 参考系；
- 请求的位置或速度类型；

定位系数块并完成插值。

伪流程：

```text
function evaluateState(bodyId, tdb):
    provider = registry.resolve(bodyId)
    segment = provider.findSegment(tdb)
    if segment not found:
        return fallbackModel(bodyId, tdb, OUT_OF_RANGE)
    state = chebyshevEvaluate(segment, tdb)
    return state with providerPrecision
```

## 14.4 轨道线

轨道线不采用固定圆或固定椭圆。

生成过程：

1. 根据目标类型确定采样时间窗口；
2. 使用自适应步长采样；
3. 曲率高、近日点或事件附近增加采样；
4. 转换到目标参考系；
5. 使用高低位或局部坐标上传 GPU；
6. 时间或参考系变化超过阈值时重新生成。

## 14.5 缓存

- 核心天体状态缓存当前时刻前后小窗口；
- 轨道采样缓存按天体、时间窗、参考系和精度键控；
- 事件扫描使用独立批量缓存；
- 不缓存跨版本结果。

# 16. 天文事件计算引擎

## 16.1 事件类型

### 角位置事件

- 合；
- 冲；
- 最大距角；
- 四分相；
- 近日点；
- 远日点。

### 盘面重叠事件

- 日食；
- 月食；
- 行星凌日；
- 掩星；
- 卫星凌越；
- 卫星食。

## 16.2 事件计算流程

```text
选择事件类型和时间范围
      ↓
粗步长扫描候选区间
      ↓
计算目标函数符号变化或极值
      ↓
数值求根 / 极值收敛
      ↓
计算接触时刻和几何参数
      ↓
生成事件记录
      ↓
与外部基准样本自动对照
```

## 16.3 主要目标函数

- 合/冲：视黄经或目标角差；
- 近日点/远日点：日心距离的一阶导数；
- 掩星/凌日：视角圆盘中心距与角半径之和或差；
- 食：本影、半影锥与目标天体相交关系。

## 16.4 光行时与视位置

事件计算按精度档位选择：

- 快速档：几何位置；
- 标准档：单次光行时修正；
- 高精度档：迭代光行时、相对速度和必要的参考系修正；
- 不以本项目结果替代专业观测预报。

## 16.5 事件结果结构

- 事件 ID；
- 类型；
- 参与天体；
- 时间范围；
- 开始、极大、结束；
- 参考系；
- 主要几何参数；
- 来源精度；
- 误差估计；
- 推荐镜头；
- 推荐时间倍率；
- 所需资源包。

## 16.6 地球表面日食表现

支持：

- 本影和半影投射到地球；
- 地球三维表面的阴影移动；
- 事件极大时刻和可见区域概览；
- 不提供基于用户定位的自动本地时区提醒。

# 17. 尺度映射系统

## 17.1 问题定义

太阳半径、行星半径、行星间距和卫星轨道存在多个数量级差异。完全统一真实比例时，绝大多数行星在太阳系全景中不可见；固定放大则会产生科学误导。

## 17.2 真实比例模式

要求：

- 单一长度比例；
- 星体尺寸和距离使用相同倍率；
- 轨道无压缩；
- 标签、轨道线和选中光标可使用屏幕空间可视增强；
- 信息区明确显示当前长度比例。

## 17.3 增强展示模式

使用分层映射：

- 恒星半径映射；
- 行星半径映射；
- 行星轨道距离映射；
- 卫星轨道距离映射；
- 小天体最小可见尺寸；
- 空间结构密度和厚度映射。

映射函数要求：

- 单调；
- 连续；
- 可逆到真实数值；
- 在 UI 中显示倍率；
- 不改变天体物理计算结果。

## 17.4 连续层级过渡

从太阳系到地表的过渡阶段：

1. 太阳系层；
2. 行星轨道层；
3. 行星系统层；
4. 卫星系统层；
5. 低轨层；
6. 地表层。

尺度映射在相机过渡中插值，避免瞬间缩放。

# 20. 全球地形与近地浏览系统

## 20.1 覆盖对象

首版全球地形系统正式支持：

- 地球；
- 月球；
- 火星。

其他固态天体可复用技术，但不强制达到同等全球覆盖。

## 20.2 地形拓扑

推荐 Cube-Sphere 六面体四叉树：

- 避免经纬度极点畸变；
- 适合统一 LOD；
- 支持六面独立流式加载；
- 可与高程和多通道材质瓦片绑定。

## 20.3 LOD 选择

依据屏幕空间几何误差：

```text
errorPixels = geometricError * projectionScale / cameraDistance
if errorPixels > threshold:
    split tile
else:
    keep tile
```

阈值随画质档位变化。

## 20.4 地形瓦片内容

- 高程；
- 颜色；
- 法线或可运行时生成法线；
- 粗糙度；
- 材质分类；
- 遮罩；
- 来源和分辨率元数据。

## 20.5 裂缝治理

- 邻接层级约束；
- 边缘裙边；
- 边界重采样；
- 法线连续；
- 父瓦片过渡混合。

## 20.6 地表相机

- 高度感知速度；
- 地形碰撞或最低安全高度；
- 地平线稳定；
- 自动近裁剪面；
- 低空模式不能穿入地表；
- 不提供人物行走或车辆控制。

## 20.7 全球基础覆盖定义

全球基础覆盖要求：

- 全球无空白区域；
- 大型地貌位置可信；
- 高空到低空连续；
- 非重点区域不额外制作影视样板；
- 最低观察高度由源数据和质量档位决定；
- 极低空微表面可以程序化，但必须保持宏观地貌不变。

# 21. 扩展太阳系空间环境

# 23. 相机、导航与操控系统

## 23.1 相机模式

### 轨道相机

围绕目标旋转、缩放和平移。

### 跟随相机

保持相对目标的稳定观察关系，允许目标继续公转和自转。

### 自由飞行相机

WASD 或等效输入控制位移，鼠标控制方向。

### 地表低空相机

适配地形高度、速度和地平线。

## 23.2 默认输入

- 鼠标左键拖动：环绕；
- 鼠标右键拖动：平移；
- 滚轮：缩放或改变速度；
- 双击目标：聚焦；
- WASD：自由飞行；
- Shift：临时加速；
- Ctrl：临时减速；
- Esc：退出巡航或纯净模式；
- Space：暂停或继续模拟时间。

正式实现需避免快捷键与浏览器默认行为冲突。

## 23.3 镜头过渡

目标切换路径分阶段：

1. 从当前局部视角退出到安全距离；
2. 切换参考系；
3. 加速前往目标；
4. 在接近目标时减速；
5. 自动设置合适 FOV 和曝光；
6. 进入目标默认观察位。

## 23.4 防穿透

- 球形或椭球体近似碰撞；
- 地形高度碰撞；
- 气态行星安全壳；
- 太阳安全壳；
- 行星环平面提示；
- 自动拉回和视觉警告。

## 23.5 动态速度

速度按当前目标半径、目标距离和场景层级确定，而不是固定米/秒。

## 27.2 真实性四级

| 标识 | 含义 |
|---|---|
| R1 | 确定性计算或可重复几何结果 |
| R2 | 公开观测数据或处理后的观测数据 |
| R3 | 科学模型和统计推演 |
| R4 | 为提升观感加入的影视增强 |

一个天体可以同时包含多个等级。例如地球地形为 R2，大气散射为 R3，强化镜头光晕为 R4。

## 27.3 内容质量

- 所有数值带单位；
- 避免超出来源精度的小数位；
- 区分平均值、瞬时值和估计值；
- 区分命名日期、发现日期和确认日期；
- 对存在争议的分类采用中性表达；
- 不使用“完全真实”“绝对精确”等不可证实措辞。

# 29. 资源分包、加载与显存治理

## 29.1 离线包结构

建议分为：

- `core-app`：前端程序、基础字体、核心 Shader；
- `core-astro`：目录、时间、核心星历；
- `core-bodies`：太阳、八大行星、月球基础资源；
- `cinematic-bodies`：S 级超高质量资源；
- `satellites-a`：A 级卫星和矮行星；
- `satellites-bc`：B/C 级资产；
- `terrain-earth`；
- `terrain-moon`；
- `terrain-mars`；
- `small-bodies`；
- `extended-space`；
- `tours`；
- `content-zh-cn`；
- `diagnostics`。

总容量不设硬上限，但每个包必须可独立校验和安装。

## 29.2 Manifest

每个资源记录：

- 逻辑 ID；
- 文件路径；
- 包 ID；
- 版本；
- 大小；
- SHA-256；
- MIME；
- 依赖；
- 质量等级；
- 适用渲染后端；
- 解码方式；
- GPU 估计内存；
- 来源记录引用。

## 29.3 加载优先级

1. 当前目标可见核心资源；
2. 相机过渡路径所需资源；
3. 当前目标近景资源；
4. 下一巡航节点；
5. 邻近系统；
6. 后台预取。

## 29.4 取消和降级

- 相机目标变化时取消过期请求；
- 高 LOD 失败时回退低 LOD；
- 高质量纹理失败时回退标准纹理；
- WebGPU 专用资源不可用时切换兼容资源；
- 不允许因为单个超高分辨率纹理失败而丢失整个天体。

## 29.5 显存预算

### 影视级模式

- 活跃 GPU 资源目标 5—6 GB；
- 预留浏览器、驱动和瞬时上传空间；
- 单个高精度目标采用分块，不一次上传全部资产；
- 同时只允许一个主目标处于最高 LOD。

### 标准模式

- 活跃 GPU 资源目标 1.5—2.5 GB；
- 关闭高成本体积效果；
- 降低地形和纹理最高 LOD；
- 小天体以点精灵和实例模型为主。

## 29.6 CPU 内存

- 解压前后峰值受控；
- 二进制解析尽量使用可转移 ArrayBuffer；
- Worker 和主线程之间避免复制；
- 资源释放后解除所有引用；
- 长稳测试监控堆内存和 GPU 资源数量。

# 30. 自动画质分级与性能策略

## 30.1 画质档位

| 档位 | 渲染后端 | 目标 |
|---|---|---|
| 影视级 | WebGPU | 1440p 60 FPS，完整效果 |
| 极高 | WebGPU | 1440p 45–60 FPS |
| 高 | WebGPU/WebGL 2 | 1080p 60 FPS |
| 标准 | WebGL 2 | 1080p 30–60 FPS |
| 安全 | WebGL 2 | 保证核心功能可用 |

## 30.2 能力检测

- GPU Adapter；
- 支持特性；
- 最大绑定、纹理和缓冲区限制；
- 压缩纹理；
- 时间戳查询；
- 分辨率和设备像素比；
- 真实基准场景结果。

不只根据 GPU 名称判断。

## 30.3 动态降级顺序

1. 降低内部渲染分辨率；
2. 降低空间粒子密度；
3. 降低阴影分辨率；
4. 降低体积云和大气采样；
5. 降低后期处理质量；
6. 降低地形 LOD；
7. 降低纹理 MIP；
8. 简化环系近景粒子；
9. 关闭次要镜头光学效果；
10. 切换简化材质。

## 30.4 迟滞机制

- 连续低于目标帧率一段时间才降级；
- 连续高于恢复阈值更长时间才升级；
- 镜头快速移动时暂时降低质量；
- 静止后逐步恢复；
- 避免每几秒反复改变档位。

## 30.5 性能监控

诊断面板显示：

- FPS；
- CPU 帧时间；
- GPU 帧时间；
- Draw Call；
- 三角形和实例数量；
- 活跃纹理和缓冲区估算；
- 当前 LOD；
- Worker 延迟；
- 加载队列。

该数据只在本地显示，不上传。

# 31. 完整离线部署方案

## 31.1 部署形态

### 单机模式

- 本地静态服务器监听 localhost；
- 浏览器打开本地地址；
- 资源在本机磁盘；
- WebGPU 可在支持浏览器的安全上下文策略下使用。

### 局域网模式

- 静态服务器部署于内网；
- 客户端通过 HTTPS 访问；
- 使用组织内部可信 CA 或受信证书；
- 所有资源同源；
- 禁止依赖公网证书在线校验作为唯一机制。

## 31.2 为什么不能使用 file://

- ES Module 加载限制；
- Fetch 和 Range 请求限制；
- Worker 和 WASM 加载限制；
- MIME 类型缺失；
- 大型资源流式加载不可控；
- WebGPU 安全上下文限制。

## 31.3 静态服务器要求

- 正确 MIME；
- HTTP Range；
- Brotli/Gzip 预压缩；
- ETag 或内容哈希；
- 静态不可变缓存；
- SPA 路由回退；
- 可配置 COOP/COEP；
- 大文件超时和并发设置；
- HTTPS 支持；
- 访问日志可关闭。

## 31.4 目录示例

```text
release/
  launcher/
  server/
  app/
  packages/
  assets/
  ephemeris/
  terrain/
  manifests/
  licenses/
  checksums/
  diagnostics/
  docs/
```

## 31.5 安装和更新

### 首次安装

1. 安装核心程序；
2. 安装核心星历；
3. 安装标准资源；
4. 可选安装影视级和地表资源；
5. 运行完整性校验；
6. 启动本地服务器；
7. 运行硬件基准测试。

### 更新

- 通过独立升级包替换资源；
- 先校验后切换 Manifest；
- 支持保留上一版本；
- 失败时回滚；
- 不在运行中覆盖正在使用的文件。

# 33. 异常处理、恢复与诊断

## 33.1 错误分类

- 启动错误；
- 资源缺失；
- 数据版本不匹配；
- WASM 初始化错误；
- Worker 崩溃；
- Shader 编译错误；
- GPU 设备丢失；
- 显存不足；
- 星历超范围；
- 巡航配置错误；
- 浏览器不兼容。

## 33.2 安全失败原则

- 不能计算高精度位置时，不输出伪高精度位置；
- 高级渲染失败时，降级而非白屏；
- 资源损坏时指出具体包；
- 关键核心数据损坏时阻止进入模拟；
- 非关键科普内容损坏时允许场景继续运行。

## 33.3 GPU 设备丢失

处理流程：

1. 暂停渲染；
2. 保存当前内存会话状态；
3. 销毁无效 GPU 句柄；
4. 重新请求设备；
5. 重建渲染管线；
6. 按当前画质重新加载必要资源；
7. 恢复场景；
8. 失败则切换 WebGL 2 或显示重启提示。

## 33.4 本地诊断包

用户可主动导出：

- 应用版本；
- 资源包版本；
- 浏览器和操作系统；
- GPU 能力；
- 最近错误；
- 性能统计；
- 加载失败文件；
- 不包含用户身份和网络外传。

导出属于故障诊断，不属于用户创作内容。

# 36. 验收标准

## 36.1 科学模拟验收

| 项目 | 验收标准 |
|---|---|
| 太阳和八大行星位置 | 在项目高精度范围内，与基准星历差异不超过 1 km |
| 月球位置 | 与基准星历差异不超过 1 km |
| S 级姿态 | 主要轴向误差不超过 0.1°，纹理经度方向正确 |
| A 级主要卫星 | 误差不超过 10 km 或来源允许误差，以较宽者为准 |
| B/C 级对象 | 满足各自数据级别，不得宣称统一高精度 |
| 日月食极大时间 | 与基准数据偏差不超过 60 秒 |
| 主要凌日事件 | 与基准数据偏差不超过 60 秒 |
| 时间倍率 | 不因帧率变化产生累计漂移 |
| 真实比例 | 几何映射误差不超过 0.01% |
| 增强比例 | UI 始终显示增强状态和倍率 |

## 36.2 视觉验收

- S 级天体拥有独立渲染方案；
- 大气无明显硬边；
- 昼夜交界连续；
- 土星环透明、投影和遮挡正确；
- 太阳远近 LOD 无明显突变；
- 地形瓦片无大面积裂缝和空洞；
- 云层不明显穿入地表；

- LOD 切换在正常观察距离不可明显感知；
- HDR 曝光切换无持续闪烁；
- R3/R4 效果在科学模式中可识别或弱化。

## 36.3 性能验收

- 2560×1440；
- RTX 4070/RX 7800 XT 级别；
- 32 GB 内存；
- 正式支持浏览器；

- 1920×1080；
- 主流集成显卡或入门独显；
- 16 GB 内存；
- 平均不低于 30 FPS；
- 搜索、时间、轨道、科普和主要事件功能完整；
- 允许关闭高成本特效。

# 39. 风险清单与规避方案

| 风险 | 级别 | 影响 | 规避措施 |
|---|---|---|---|
| “影视级”目标无限扩张 | 高 | 周期和成本失控 | 建立 S/A/B/C 资产验收标准和镜头距离边界 |
| 全部卫星数据质量不一致 | 高 | 无法统一高精度 | 使用 P0—P4 分级，显示数据覆盖和不确定性 |
| WebGPU 浏览器差异 | 高 | 设备不可用或渲染错误 | WebGL 2 降级、版本冻结、驱动黑名单 |
| 内网 HTTP 无 WebGPU | 高 | 影视级模式无法启动 | localhost 或内网 HTTPS，启动时检测说明 |
| 全球地形资源巨大 | 高 | 安装和加载困难 | 分包、瓦片、Range、LOD、按需加载 |
| 显存峰值失控 | 高 | 浏览器崩溃 | 预算、请求取消、LRU、单主目标最高 LOD |
| 自研 Shader 维护复杂 | 高 | 多后端不一致 | 参数语义统一、视觉回归、后端适配层 |
| 未来闰秒未知 | 中 | 未来 UTC 时间偏差 | 内部连续时间尺度、预测标识、数据包更新 |
| 事件算法误差 | 高 | 科学可信度下降 | 基准样本、数值收敛测试、精度分级 |
| 公开数据许可差异 | 高 | 无法分发 | 逐数据集台账，不把“公开可访问”等同于可任意使用 |
| NASA 页面含第三方素材 | 中 | 版权风险 | 只使用明确标识来源的数据，检查第三方版权说明 |
| 真实尺度体验不佳 | 中 | 用户认为系统不可用 | 增强模式默认、倍率透明、一键真实比例 |
| 程序化资产被误解为真实 | 高 | 科学误导 | 强制 R3/R4 标识，内容审核 |
| 浏览器内存回收不可控 | 高 | 长稳失败 | 主动释放 GPU 资源、分块加载、长期监控 |
| Firefox 能力变化 | 中 | 兼容结果不稳定 | 定义为兼容支持，不作为影视级正式基线 |
| 无后端导致更新不便 | 中 | 数据更新依赖完整发布 | 独立静态数据包和 Manifest 版本化 |
| 无本地状态影响体验 | 低 | 每次重置 | 明确产品决定，提供合理默认状态 |
| 无正式科学专家审核 | 中 | 无法声称权威认证 | 自动对照、来源可追溯、保守表述 |

# 43. 数据来源、版权与署名规范

## 43.3 NASA 内容使用注意事项

NASA 内容通常可用于教育、信息展示、计算机图形模拟和网页，但：

- NASA 标志和识别符不属于普通公共领域素材；
- 不得暗示 NASA 对项目背书；
- NASA 页面可能包含第三方版权素材；
- 必须逐项检查素材标识；
- 应在项目来源说明中进行事实性署名；
- 项目不得使用 NASA 名义为自身准确性背书。

# 结论

本项目应采用以下最终架构基线：

> **React 应用层 + 原生 Three.js 引擎层 + WebGPU/WebGL 2 双渲染后端 + Rust/WebAssembly 天文内核 + Web Worker 并行计算 + JPL/NAIF 星历构建管线 + 分层参考系与浮动原点 + Cube-Sphere 全球地形 + S/A/B/C 资产体系 + 完整离线静态资源包。**

项目的最大挑战不是绘制八个行星，而是同时控制天文精度、影视资产、超大尺度、地表 LOD、浏览器显存、数据许可和双渲染后端。必须严格按照技术门禁分阶段推进，先验证星历、坐标、双后端和资源治理，再扩大资产范围。
